如何创建只能在堆或栈上的对象

发表于 2021-06-27 分类于 编程语言 ， C++

如何创建只能在堆或栈上的对象

在C++中，类的对象建立分为了两种，一种是静态建立，例如A a;，一种是动态建立，例如A* ptr= new A;

静态建立一个类对象，是由编译器为对象在栈中分配内存，是通过在栈帧中分配适当的栈空间，然后在栈内存中调用构造函数形成一个栈对象。

动态建立一个类对象，是使用new运算符将对象建立在堆空间中。首先执行operator new()函数，在堆空间中搜索适合的内存并进行分配。然后调用构造函数初始化内存空间。

只在栈上分配类对象

只有使用new运算符，对象才会建立在堆上。因此首先需要先禁用new运算符。new运算符是C++内建的，执行new operator总是先调用的operator new，因此我们需要对new操作符进行重载，并申请为private，就可以保证不能使用new实例化对象：

如下图，在栈上建立对象可以，但是想使用new在堆上建立对象就会报错operator new和operator delete都是类的私有成员函数，无法访问。

只在堆上分配类对象

只能在堆上建立，那就是等同于不能再栈上建立。那么就先分析对象建立在栈上的时候，需要哪些机制进行保证，然后破坏某些机制，对象就不能在栈上建立，那么就只能在堆上建立。

首先说一下对象建立在栈上的时候，由编译器进行内存的分配。首先调用构造函数来构造栈对象，当对象使用完之后，或者说，对象已经超出了某个函数栈帧的生命周期，就会调用析构函数来释放栈对象所占有的空间。也就是说，编译器管理了对象的整个生命周期。

如果编译器无法调用类的析构函数。例如，类的析构函数是私有的，那么编译器无法调用析构函数来释放内存。因此，编译器在为类对象分配栈空间的时候，需要先检查类的析构函数是否可以被访问。其实不光是检查析构函数，只有是非静态的函数，编译器都会进行检查。如果类的析构函数是私有的，则编译器不会在栈空间上为类分配内存。

最后得出结论，将类的析构函数设为私有，类对象就无法建立在栈上。

使用A a;语句来建立对象，编译会报错，提示析构函数无法访问。那么就只能通过new操作符来建立对象。构造函数是公有的，因此可以直接调用。类中必须提供一个destroy()函数，来对内存空间进行释放。类对象使用完毕后，也必须调用destroy()释放内存。

上述方法存在缺点：

  1. 无法解决继承问题

    如果把ObjOnHeap设置成基类，则析构函数通常需要设置成virtual，然后在子类中重写，以实现多态。那么这个地方不能就析构函数设置为private。

    可以将析构函数设置成protected，则在类外无法访问protected成员，子类可以访问。

    PS：如果delete一个动态分配的对象的指针，会执行虚构函数。如果指针指向的是整个继承体系中的某一个类型，则可能出现指针的静态类型与被删除对象的动态类型不符合的情况。

    因此，就通常把基类的析构函数设置为虚析构函数，就可以保证执行的是正确的析构函数版本

  2. 类的使用不方便，使用new创建对象，但是使用destroy函数释放对象，不能使用delete

    使用delete会报错，因为delete对象的指针，会调用对象的析构函数，但是这个地方无法访问析构函数。

为了解决以上两个问题，可以把构造函数设置成protected，然后提供一个public的static函数来完成构造，这样不使用new，而是通过一个函数来构造，使用一个函数来进行析构，有点类似于单例模式：

这样，就通过create函数在堆上创建类对象，调用delete函数释放内存。

简单工厂模式

结构组成

简单工厂模式结构组成通常包括：

  1. 工厂类：工厂模式的核心类，定义了一个用于创建指定的具体实例对象的接口

抽象工厂模式

结构组成

抽象工厂模式的结构组成和工厂方法模式一样：

  4. 具体产品类：具体工厂所创建的对象

  • 工厂方法模式和抽象工厂模式，都需要增加一个对应的产品的具体工厂类

单例模式C++的实现

单例模式是一种创建型设计模式，保证一个类仅有一个实例，并且提供一个访问实例的全局访问点，该实例被所有程序模块共享。

为什么需要控制一个类只有一个实例呢？常见的原因是为了控制对于一些共享资源的访问，例如之前实现的TinyWebserver里面的数据库连接池、都是采用单例模式来实现的。如果已经创建了一个实例，那么在后面调用想要再次创建一个新的实例时，会获取到之前已经创建的对象，而不是一个新的对象。使用普通的构造函数无法这样来实现，因此需要将构造函数私有化。

创建全局唯一的对象后，还需要提供一个对于实例的全局访问点。

什么是单例模式

定义一个单例类：

  1. 私有化构造，防止外界可以创建单例类的对象
  2. 使用静态构建方法，这个方法会调用私有构造函数来创建对象，并保存到一个静态成员变量中。
  3. 使用一个公有的静态方法获取实例

单例模式适合使用的场景

  • 如果程序的某个类对于所有的其他模块，只有一个可用的实例，可以使用单例模式，例如一个数据库连接池、一个日志模块。
  • 如果需要对于一些全局变量进行严格的控制，也可以使用单例模式。

单例模式的优缺点

优点：

  • 保证了一个类只有一个实例
  • 提供了指向该实例的全局访问节点
  • 在首次请求单例模式的时候会初始化，其后都是获取实例

缺点：

  • 多线程环境下需要进行特殊处理，避免多个线程多次创建单例对象
  • 进行单元测试会比较麻烦，因为很多测试框架都是基于继承的方式来创建模拟对象。

单例模式C++的一些实现

总体的实现方式如下：

  1. 在类中添加一个私有静态成员变量用于保存单例的实例
  2. 声明一个公有静态构建方法用于获取单例实例
  3. 在静态方法中实现”延迟初始化“。该方法在首次被调用的时候，会创建一个对象，并存储在静态成员变量中。此后的调用都会返回该实例
  4. 将类的构造函数设为私有。类的静态方法仍然可以调用构造函数，但是其他对象不能调用
  5. 写代码的时候注意，将单例的构造函数的调用替换为对其静态方法的调用。

懒汉版

第一版 懒汉实现

懒汉版，单例实例在第一次被使用的时候才进行初始化，也叫做延迟初始化。

第二版 懒汉+解决内存泄漏

上面方式存在内存泄漏的问题，因为当程序运行结束，没有对全局的单例对象进行析构，解决方法有：

  1. 智能指针，获取单例的时候使用指针指针，当指针计数为0的时候，就释放对象
  2. 使用静态的嵌套类对象

在程序运行结束时，会调用静态成员deletor的析构函数，析构函数就会删除单例的唯一实例。

第三版 多线程安全=双检测锁

经典的一款实现，就是采用双检测锁模式

第四版 多线程安全=C++11内部静态变量

C++11规定了local static在多线程条件下的初始化行为，要求编译器保证内部静态变量的线程安全性。在C++11标准下，《Effective C++》提出了一种更加优雅的单例模式实现，使用函数内的local static对象，这样只有在第一次访问getInstance方法的时候才会创建实例。

饿汉版

饿汉版指的就是：单例实例在程序运行的时候就会被立即执行初始化

在main函数之前进行初始化，不用考虑线程安全的问题。但是潜在的一个问题就是：no-local static对象（函数外的static对象）在不同的编译单元中的初始化顺序未定义。也就是说：static Singleton instance; 和 static Singleton& getInstance() 二者的初始化顺序不确定。如果前者未初始化完成，调用getInstance会返回一个未定义的实例。

总结

先是对单例模式的定义进行了说明，然后简单说了一下单例模式适合用于哪些场景，分析单例模式的一些优缺点，再使用C++主要两种版本：懒汉版和饿汉版来实现了一遍单例模式，以及从线程安全角度考虑需要注意哪些方面。

B+树总结

m阶B+树定义：

  1. 每个节点最多有m-1个子节点
  2. 除根节点外，每个节点至少有m/2个子节点，根节点至少有两个子节点
  3. 有k个子节点的非叶子节点有k个键

如图是一个3阶的B+树

如果将B+树应用到数据库，所有的叶子节点除了保存关键字外，还应该有指向关键字记录的指针，也可以看到叶子节点按照关键字从小到大进行了链接。

查找

B+树中，所有的内部节点相当于叶子节点的索引，所有的关键字被存储在叶子节点中，因此，所有从根节点出发做查找操作，如果内部节点的关键字值等于查找关键字，并不算真正的查找完毕，还需要继续向下直到叶子节点。

而且，无论查找成功与否，每次查找操作都是进行了一次从根节点到叶子节点的路径。

插入

插入首先需要寻找到插入的位置，B+树真正的插入是在叶子节点中进行，并且需要保持叶子节点内部关键字的排序。插入叶子节点，如果导致节点的关键字个数超过了阶数，需要对节点进行分裂。而且插入的关键字是叶子节点内部关键字的最值，那么还需要对索引进行修改，也就是修改父节点。

279 完全平方数

首先还是确认，这可以是一个完全背包问题。物品是无限多个完全平方数：1、4、9、16等等，需要求凑满背包的最小完全平方数数量

思路

  1. 确认dp[i]：凑齐i需要的最少完全平方数数量
  2. 确认递推公式：dp[i] = min(dp[i], dp[i - n^2] + 1)
  3. 初始化dp数组：dp[0] =0，其余全部初始化为INT_MAX
  4. 确认遍历顺序：完全背包，背包容量从小到大，背包容量和物品无所谓嵌套顺序，个人喜欢先遍历物品
  5. 手动模拟（再强调多次也可以，一定要手动模拟）

注意

  • 无论是先遍历背包还是先遍历物品都是可以的
  • 第二个循环是从j = i * i开始的

进程间通信方式

发表于 2021-06-21 分类于 操作系统

进程间的通信方式

  1. 管道/匿名管道（Pipes）：用于具有亲缘关系的父子进程间或者兄弟进程间的通信，但是通信是单向的，如果要实现全双工，需要两边都开两个管道文件描述符

    通过匿名管道实现进程间通信的步骤如下：

    1. 父进程创建管道，得到两个文件描述符指向管道的两端
    2. 父进程fork出来子进程，子进程也有两个文件描述符指向同一管道，因为父子进程共享文件描述符
    3. 父进程关闭fd[0]，子进程关闭fd[1]，即父进程关闭管道读端，子进程关闭管道写端，这样父进程可以往管道中写，子进程可以从管道中读。

    一个简单的使用管道的例子，这是使用的半双工通信，如果要进行全双工的通信，需要使用两对文件描述符

  2. 有名管道（Names Pipes）：

    匿名管道实际上只能用于父子进程间的通信，为了克服这个缺点，提出了有名管道。有名管道实际上是提供了一个路径名与之关联，因此，创建有名管道的进程哪怕是不存在亲缘关系的进程，只要可以访问该路径，就可以通过有名管道进行通信。

    我用多进程写，总是写不好，这个地方先不放代码了，可以参加一下两个博客，都是分别读和写实现的

    • Linux C编程 有名管道FIFO
    • 有名管道实现进程间通信

  3. 信号（Signal）：信号是由用户、系统或者进程发送给目标进程的信息，以通知目标进程某个状态的改变或者系统异常。

    比如前台进程使用Ctrl + C结束进程、系统发生了异常（浮点异常或者非法内存访问）、系统状态变化（alarm时钟）、kill命令或调用。在服务器程序中，有一种统一事件源的做法。通常是信号处理函数接受信号，然后把信号值传递给主循环，主循环根据信号值来调用对应的逻辑代码。例如，信号处理函数捕获信号后，把信号值传递通过管道传递给主循环，主循环通过IO复用调用来监听管道上的可读事件，就可以实现信号事件和其他IO事件一样的处理方式了。

  4. 消息队列（Message Queuing）：用于在进程间传递二进制数据，每个数据块都有一个特定的数据类型，接收方可以根据类型有选择地接受数据

    消息队列涉及到的一些系统调用，粗浅说一下：

    • 使用msgget创建或者获取一个消息队列，同样有一个全局唯一的键值，创建的过程会对应创建内核数据结构
    • msgsnd将一个消息添加到消息队列中，添加的消息包含消息类型和消息数据两个部分
    • msgrcv从消息队列中获取消息

    初次之外，还有一个msgctl控制相关属性

  5. 信号量（Semaphores）：用于在多个进程对于临界资源的访问进行控制。

    使用信号量，存在如下的一些系统调用：

    • semget：创建或者获取一个信号量集，同样需要一个键值，用于全局唯一标记一个信号量集。创建信号量集会产生相应的内核数据结构。通常对于这种内核数据结构，都是包含：权限相关信息、时间相关信息
    • semop：对信号集中的变量执行PV操作，实际上就是对内核数据进行操作，可以对信号量集中的具体信号进行操作（通过传入指针的方式）。
    • semctl：对信号量进行直接控制，例如对于某个信号量集中的特定信号量进行数据提取、移除、获取内核数据结构相关信息等等

  6. 共享内存（Shared Memories）：可以让多个进程同时访问同一块内存空间，由于不涉及到数据的传输，因此是一个高效的IPC机制。但是共享内存需要结合一些同步措施来对共享内存进行访问，否则会出现竟态条件。

    共享内存设计到的一些系统调用，同样粗浅说一下：

    • 使用shmget获取一段共享内存，有一个键值参数可以保证共享内存全局唯一，通过这个值不同进程把共享内存附加到自己的地址空间

    • 使用shmat将共享内存附加到进程的地址空间中

    • 使用完毕后，调用shmdt将共享内存从地址空间分离

    其中，还可以使用shmctl系统调用来对共享内存的属性进行配置

线程间的同步的方式

线程间同步的方式主要有三种：信号量、互斥量（互斥锁）和条件变量

  • 信号量类似于多进程的信号量
  • 互斥量，或者说互斥锁，保护关键代码段，以确保线程的独占式访问
  • 条件变量，用于线程之间同步共享数据的值，也提供了一种通知机制，当某个共享数据达到某个值的时候，唤醒等待这个共享数据的线程

POSIX信号量

POSIX信号量和进程间通信的System V IPC信号量是不同的，接口比较相似

常用的信号量操作方式：初始化信号量、销毁信号量、PV操作之类，如以下API

初始化和销毁信号量都要小心，防止初始化已经初始化过的信号量、防止销毁正在有进程使用的信号量

互斥量

或者说，互斥锁，用来保护关键代码段，以确保独占式的访问，类似于二进制信号量。

对于互斥量，也可以设置是否允许进程间共享、只能一个进程的多个线程内共享

条件变量

用于线程之间共享同步数据的量，当共享数据达到某个值的时候，唤醒这个等待共享数据的线程
