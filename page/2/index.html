12 矩阵中的路径

题目

给定一个m x n二维字符网格board和一个字符串单词word。如果word存在于网格中，返回 true ；否则，返回 false 。

单词必须按照字母顺序，通过相邻的单元格内的字母构成，其中“相邻”单元格是那些水平相邻或垂直相邻的单元格。同一个单元格内的字母不允许被重复使用。

思路

这道题使用的递归，递归的思路：

  • 递归参数

    当前元素在矩阵中的行和列索引、已经匹配的长度k

  • 中止条件

    • 返回false：（1）当前索引越界 （2）当前矩阵的元素与目标字符不一致（3）矩阵元素已经访问过
    • 返回true：k = len(word) - 1，说明已经全部匹配

  • 递推工作

    1. 标记当前矩阵元素：board\[i][j] = ' '，代表已经访问过，防止以后的重复搜索

    2. 搜索下一单元格：上下左右四个方向开启递归，并记录结果

    3. 还原当前的矩阵：将board[i][j]还原到初始值，也就是word[k]

      Cm：递归初始参数的设定，因为是已经匹配了长度k，那么word[k]就是当前匹配成功的元素

  • 返回值

    返回bool变量res，代表是否搜索成功

注意

  • 函数传参，该引用的时候引用，可以极大提高空间和时间复杂度
  • 明确递归函数的定义和变量的定义

复杂度分析

  • 时间复杂度：O(3^KMN)，最差的情况就是遍历所有的元素一共MN个，每一个都匹配K次，每次有三个方向的选择
  • 空间复杂度：O(K)

两种高效的时间处理模式

发表于 2021-06-20 更新于 2021-06-21 分类于 计算机网络

两种高效的事件处理模式

服务器程序上通常需要处理三类事件：IO事件、信号、定时事件。那么对于这三类如何进行处理，有两种高效的事件处理模式：Reactor和Proactor

通常使用同步IO模型来实现Reactor模式，

0x01 Reactor模式

总述

Reactor使用主线程（IO处理单元）只负责监听文件描述上是否有事件发生，如果有就立刻通知工作线程（逻辑单元），除此之外，主线程不做其他的事情。读写数据、接受连接，以及处理用户请求都是在工作线程中完成。

实现

使用同步IO模型（例如epoll_wait），实现Reactor模式的流程就是：

  1. 主线程往epoll_wait内核事件表中注册socket上的读就绪事件
  2. 主线程调用epoll_wait等待socket上有数据可读
  3. 当socket可读的时候，epoll_wait通知主线程，主线程将可读事件放入到请求队列中
  4. 睡眠在请求队列上的某个工作线程被唤醒，从socket中读取数据，处理用户请求，然后往epoll内核事件表中注册socket上的写就绪事件
  5. 主线程调用epoll_wait等待socket可写
  6. 当socket可写的时候，epoll_wait通知主线程，主线程把socket可写事件放入请求队列
  7. 睡眠在请求队列的某个工作线程被唤醒，往socket中写入之前服务器处理好的用户请求结果

0x02 Proactor模式

总述

Proactor模式将所有的IO操作交给主线程和内核进行处理，工作线程仅仅负责业务逻辑，从某种程度上来说，Proactor模式更加适合广义的服务器编程框架

实现

使用同步IO（epoll_wait）模拟Proactor模式的工作流程如下：

  1. 主线程往epoll_wait内核事件表中注册socket上的读就绪事件
  2. 主线程调用epoll_wait等待socket上有数据可读
  3. 当socket上有数据可读时，epoll_wait通知主线程，主线程从socket中循环读取数据，直到数据被读取完毕，然后将读取到的数据封装称一个请求对象插入到请求队列中
  4. 睡眠在请求队列上的某个工作线程被唤醒，它获得请求对象并处理客户请求，然后往epoll_wait内核事件表中注册socket的写就绪事件
  5. 主线程调用epoll_wait等待socket可写
  6. 当socket可写的时候，epoll_wait通知主线程，主线程往socket中写入服务器处理客户请求的结果

小结

这个是对事件的处理方式。

一个程序是从main函数开始执行么

发表于 2021-06-18 分类于 操作系统

一个程序是从main函数开始执行么

不是的

  • 一个程序开始运行，实际上是操作系统为它创建了一个进程后，把控制权交到了入口函数Entry Point，入口函数也往往是运行库的一部分。
  • 入口函数对运行库和程序运行环境进行初始化，包括堆栈、IO、线程、全局变量的构造等等。
  • 入口函数初始化之后，调用main函数，并且传入main函数需要的参数，argc、argv字符串数组
  • main函数执行完毕之后，实际上就是函数调用完毕，返回到入口函数，入口函数进行清理工作，例如全局变量的析构、堆销毁、关闭IO等，然后进行系统调用结束进程

额外的一些补充，glibc的入口是_start，我们也可以在编译的时候来重写_start函数，重写了之后原来_start给main函数准备的退出清理工作就没了，这个时候main函数的退出也不能使用return，而是要使用exit。同样编译的使用也可以使用-e属性，或者是-nostartfiles，具体就不展开说了，参考链接

_start函数会调用__libc_start_main函数，大概的基本流程如下，可以总结为，获取参数列表和环境变量，为下一步调用准备：

exit函数也是有讲究的，不过根本就是使用系统调用结束进程

IO复用

发表于 2021-06-17 分类于 计算机网络

IO复用的简单描述

IO复用的优点就是程序可以使用不同的措施例如select、poll、epoll系统调用，监听多个文件描述符。需要注意的是，IO复用虽然可以监听多个文件描述符，但是使用这三个系统调用本身实际上还是阻塞的，如果不采用类似多线程/多进程的方式对满足条件（可读、可写等）的文件描述符进行处理，在单个程序中实际上还是对这些满足条件的文件描述符顺序进行处理的。

Linux下实现IO复用主要是通过select、poll、epoll三个系统调用。

select

简要描述

select系统调用的用途是：在指定的一段时间内，监听用户感兴趣的文件描述符上，发生的可读、可写和异常等事件。可读和可写两个属性很好理解，但是目前异常属性只有两种条件：

  1. 某个套接字收到了带外数据的到达
  2. 某个已经置为分组模式的伪终端存在并可以从主终端读取的控制状态信息（懵了，不理解）

API说明

通过API可以更加直观的理解

  • maxfdp1：指定被监听的文件描述符的总数。通常被设置为select监听的所有文件描述符中的最大值+1，因为文件描述符从0开始计数。
  • readset、writeset、exceptset分别指向可读、可写和异常对应的文件描述符集合，传入的指针类型，select返回的时候内核会对这三个集合进行修改，使得应用程序知道哪些文件描述符是就绪的。
  • timeout：设置select的超时时间，采用指针类型方便内核对其进行修改，告诉应用程序select等待了多久。

结果说明

对于select系统调用执行的结果：

  • select执行成功的话，返回就绪（可读、可写和异常）文件描述符的总数，应用程序就可以获得三类满足条件的文件描述符集合，就看怎么进行处理。
  • 如果在超时时间内没有任何文件描述符准备就绪，返回0。
  • 如果执行失败，返回-1并且设定errno
  • 如果在select的等待期间，程序接收到信号，则返回-1，并且设定errno为EINTR

epoll

epoll是Linux上特有的IO复用函数，select和poll在windows系统中也有实现，但是epoll是独此一家。

select和poll都使用一个系统调用函数就能完成任务，但是epoll稍微复杂一些，需要使用多个函数来完成任务。

1. epoll的事件表

epoll将用户关心的文件描述符上的事件放入到内核的一个事件表中，就不用像select和poll，每次调用都需要重复传入文件描述符集合，epoll使用了一个特殊的文件描述符来对这个事件表进行标识，使用epoll_create来进行创建

size只是提示内核这个事件表需要多大，返回的文件描述符用于作为其他接下来使用的epoll系统调用的第一个参数，以指定访问的内核事件表。

2. 控制事件表

  • epfd：也就是内核事件表的文件描述符

  • op和fd：要操作的文件描述符以及操作类型

    • EPOLL_CTL_ADD：往事件表中注册fd上的事件
    • EPOLL_CTL_MOD：修改fd上的注册事件
    • EPOLL_CTL_DEL：删除fd上的注册事件

  • event：事件类型，poll的事件类型epoll基本上都支持，在poll的事件类型宏前加上E则变成epoll支持的事件类型了。

3. 真正的调用epoll_wait

epoll系统调用的主要接口就是epoll_wait，用于在一段超时时间内等待一组文件描述符上的事件，原型如下：

  • events：如果监测到有事件，则将所有就绪的事件从内核事件表复制到events数组中。
